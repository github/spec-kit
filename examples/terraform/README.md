# Terraform Walkthrough: `s3-secure-bucket`

This directory shows a complete, end-to-end run of the InfraKit Terraform workflow against a
fictional platform-engineering team at Northwind Logistics. The deliverable is a reusable AWS
S3 bucket module that ships the Northwind security baseline.

Read this if you want to see, in one place, every file InfraKit produces when you take a
Terraform module from "I need an S3 bucket" to "module v1.0.0 is ready to publish."

---

## Scenario

Northwind's `platform` team is publishing a new module to its `infra-modules` repo. The module
will be consumed by 5 product teams (orders, fulfillment, returns, analytics, customer-portal)
for application data buckets. The audit and security teams require:

- Encryption at rest with a customer-managed KMS key.
- All four `block_public_*` flags set.
- TLS-only access.
- Versioning and lifecycle on non-current versions.
- Cross-region replication for prod, gated to prod only.

The platform team kicks off the workflow with:

```bash
/infrakit:setup                    # one-time: configure context, coding style, tagging standard
/infrakit:create_terraform_code    # creates the spec via the Cloud Solutions Engineer → Architect → Security workflow
/infrakit:plan <track-name>        # generates plan.md + tasks.md
/infrakit:implement <track-name>   # writes the .tf files
/infrakit:analyze <track-name>     # cross-artifact consistency
/infrakit:architect-review <track> # architecture quality gate
/infrakit:security-review <track>  # compliance audit (SOC 2, PCI-DSS)
/infrakit:review modules/s3-secure-bucket  # code review of the HCL itself
```

## How to read it

1. Start with [`.infrakit/context.md`](.infrakit/context.md) — the project constitution. Every command reads this.
2. Read [`.infrakit/coding-style.md`](.infrakit/coding-style.md) and [`.infrakit/tagging-standard.md`](.infrakit/tagging-standard.md). These are non-negotiable for generated code.
3. Then walk the track artifacts in order:
   - [`spec.md`](.infrakit_tracks/tracks/s3-secure-bucket-20260415-093000/spec.md) — what we're building, why, and the acceptance criteria.
   - [`plan.md`](.infrakit_tracks/tracks/s3-secure-bucket-20260415-093000/plan.md) — the Terraform-engineer's blueprint, with every resource and variable nailed down.
   - [`tasks.md`](.infrakit_tracks/tracks/s3-secure-bucket-20260415-093000/tasks.md) — the implementation checklist (already checked off in this example).
   - [`analyze.md`](.infrakit_tracks/tracks/s3-secure-bucket-20260415-093000/analyze.md) — spec/plan/code consistency check.
   - [`architect-review.md`](.infrakit_tracks/tracks/s3-secure-bucket-20260415-093000/architect-review.md) — cost, reliability, completeness review.
   - [`security-review.md`](.infrakit_tracks/tracks/s3-secure-bucket-20260415-093000/security-review.md) — SOC 2 + PCI-DSS audit.
   - [`review.md`](.infrakit_tracks/tracks/s3-secure-bucket-20260415-093000/review.md) — final code review of the HCL.
4. Finally, look at the produced [module](modules/s3-secure-bucket/) — the deliverable.

## Trying it locally

The module is real Terraform. From this directory:

```bash
cd modules/s3-secure-bucket
tofu init -backend=false
tofu validate
tofu fmt -check
```

All three should pass. The module is not directly applicable on its own (it has no `provider`
block — that's by design; live configs supply providers). To plan against it:

```hcl
# /tmp/try-it/main.tf
provider "aws" { region = "us-east-1" }

module "test" {
  source = "/path/to/examples/terraform/modules/s3-secure-bucket"

  name                = "test"
  environment         = "dev"
  team                = "platform"
  cost_center         = "CC-0001"
  data_classification = "internal"
}
```

Then `tofu init && tofu plan` (you'll need real AWS credentials to plan against a real account).
