# Crossplane Walkthrough: `XPostgreSQLInstance`

This directory shows a complete, end-to-end run of the InfraKit Crossplane workflow against a
fictional platform-engineering team at Acme Corp. The deliverable is a reusable Crossplane
composite resource (`XPostgreSQLInstance`) that wraps AWS RDS PostgreSQL via the
`upbound/provider-aws-rds` provider.

Read this if you want to see, in one place, every file InfraKit produces when you take a
Crossplane composition from "I need a managed Postgres" to "Composition `v1.0.0-rc.1` is ready
to publish."

---

## Scenario

Acme Corp's `platform` team is publishing a new composition to its `infra-platform` monorepo.
The composition will be consumed by 4 product teams (payments, data, frontend, mobile) for
application database needs. The audit and security teams require:

- Storage encrypted at rest with a per-instance customer-managed KMS key.
- No public network access ever (non-overridable).
- Multi-AZ and deletion protection default `true` in prod, `false` elsewhere; both overridable.
- All managed resources tagged with the required keys from `.infrakit/tagging-standard.md`.
- Connection details published to a Kubernetes Secret in the claiming namespace.

The platform team kicks off the workflow with:

```bash
/infrakit:setup                       # one-time: configure context, coding style, tagging standard
/infrakit:new_composition             # creates the spec via Solutions → Architect → Security workflow
/infrakit:plan <track-name>           # generates plan.md + tasks.md
/infrakit:implement <track-name>      # writes definition.yaml + composition.yaml + examples
/infrakit:analyze <track-name>        # cross-artifact consistency
/infrakit:architect-review <track>    # architecture quality gate
/infrakit:security-review <track>     # compliance audit (SOC 2, PCI-DSS)
/infrakit:review compositions/xpostgresql-instance  # YAML review
```

## How to read it

1. Start with [`.infrakit/context.md`](.infrakit/context.md) — the project constitution. Every command reads this.
2. Read [`.infrakit/coding-style.md`](.infrakit/coding-style.md) and [`.infrakit/tagging-standard.md`](.infrakit/tagging-standard.md). These are non-negotiable for generated YAML.
3. Walk the track artifacts in order:
   - [`spec.md`](.infrakit_tracks/tracks/xpostgresql-instance-20260415-101500/spec.md) — what we're building, why, and the acceptance criteria.
   - [`plan.md`](.infrakit_tracks/tracks/xpostgresql-instance-20260415-101500/plan.md) — the Crossplane-engineer's blueprint with verified API versions and per-resource patch lists.
   - [`tasks.md`](.infrakit_tracks/tracks/xpostgresql-instance-20260415-101500/tasks.md) — implementation checklist (already checked off).
   - [`analyze.md`](.infrakit_tracks/tracks/xpostgresql-instance-20260415-101500/analyze.md) — spec/plan/composition consistency check.
   - [`architect-review.md`](.infrakit_tracks/tracks/xpostgresql-instance-20260415-101500/architect-review.md) — cost, reliability, completeness review.
   - [`security-review.md`](.infrakit_tracks/tracks/xpostgresql-instance-20260415-101500/security-review.md) — SOC 2 + PCI-DSS audit.
   - [`review.md`](.infrakit_tracks/tracks/xpostgresql-instance-20260415-101500/review.md) — final YAML review.
4. Finally, look at the produced [composition](compositions/xpostgresql-instance/) — the deliverable.

## Trying it locally

The YAML is real Crossplane. From this directory:

```bash
# Syntactic validation (no cluster needed):
python3 -c 'import yaml,sys; [list(yaml.safe_load_all(open(f))) for f in sys.argv[1:]]' \
  compositions/xpostgresql-instance/definition.yaml \
  compositions/xpostgresql-instance/composition.yaml \
  compositions/xpostgresql-instance/examples/claim-dev.yaml \
  compositions/xpostgresql-instance/examples/claim-prod.yaml

# Full render (needs the crossplane CLI and function image cached):
crossplane render \
  compositions/xpostgresql-instance/examples/claim-dev.yaml \
  compositions/xpostgresql-instance/composition.yaml \
  functions.yaml
```

To apply against a real cluster you'll need Crossplane v1.15+, the `upbound/provider-aws-rds` and
`upbound/provider-aws-kms` providers v1.2.0+, and the `function-patch-and-transform` v0.6.0
function installed.
