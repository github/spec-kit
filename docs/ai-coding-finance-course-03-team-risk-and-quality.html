第 3 期：让不同 AI 工具在同一套团队规则下工作

理解重复造轮子、架构越界和验证不足从哪里来，以及知识、记忆、证据和人工审核如何共同控制风险。

Course 031 / 13
The New Risk

功能实现成本下降，团队风险反而更容易被放大

改得太快

没有充分理解项目，就直接生成大量差异。

看得不全

只读局部文件，错过已有能力和真实边界。

做法不一

不同工具按各自理解实现同一需求。

证据不足

能运行一次就宣布完成，失败路径无人验证。

Course 032 / 13
Failure 01

重复造轮子：没有先找到项目已有能力

做法会发生什么判断标准
直接新增上传状态服务形成两套状态语义和两套维护入口先查源码、Code Graph、历史 PR 和模块说明
复用现有状态能力改动集中，调用方保持一致说明复用了哪个类、接口或配置
现有能力确实不够提出接口需求并由负责人评审不要绕开边界修改别的模块内部
Course 033 / 13
Failure 02

架构越界：为了局部功能移动了公共边界

公共接口

新增模糊参数、改变含义或破坏兼容性，会影响所有调用方。

跨模块

直接改其他模块内部或访问其数据库，会打破责任边界。

配置

新增配置默认行为与上一版不同，会让升级用户意外改变行为。

停下信号

出现以上变化时，AI 必须给出影响范围并等待人类决策。

Course 034 / 13
Failure 03

验证不足：只证明正常路径，不证明真实场景

验证能证明什么还需要追问什么
小文件上传成功只证明最简单路径仍缺大文件、超时和外部存储异常
单元测试通过证明局部逻辑仍缺模块契约和真实环境差异
完整证据包正常、失败、边界和未覆盖项都清楚审核人可以判断剩余风险
Course 035 / 13
Unified Entry

工具可以不同，但进入项目后看到的规则必须一致

Codex、Claude Code、Cursor Agent、Trae 的界面不同，项目规则、任务上下文和停下条件应由仓库统一提供。

统一规则

模块边界、公共接口、测试和提交要求进入工具可读取的入口。

统一任务

同一个 issue、上下文包和计划，不因换工具而重新解释。

统一证据

不管谁生成代码，都用相同测试与审核标准判断。

目标不是让所有人使用同一种 AI，而是让不同 AI 在同一个真实项目约束下协作。
Course 036 / 13
Knowledge Layer

共享知识让 AI 少猜、少重复、少另起炉灶

源代码

当前系统行为的第一事实。

Code Graph

类、接口、调用和模块依赖的快速投影。

项目说明

目录用途、模块职责、运行方法和架构原则。

历史决定

解释为什么选择当前设计，避免反复讨论已定事项。

问题与变更

issue 和 PR 记录需求、讨论、证据和责任。

术语与规范

统一团队对同一个业务词和技术边界的理解。

Course 037 / 13
Memory Layer

一次失败只有被整理和复用，才会变成团队经验

不是把聊天记录全部上传，而是从失败中提炼以后能直接帮助判断的内容。

1发生测试失败、评审退回、线上事故或重复返工。
2复盘区分事实、根因、误判和环境因素。
3提炼形成判断条件、检查项、样例或反例。
4审核去除敏感信息，确认适用范围。
5复用进入后续任务的规则、知识、测试或记忆检索。
Course 038 / 13
Memory Preview

个人记录、团队经验和正式指导不是同一种东西

层次适合保存什么基本边界
个人工作记录本轮尝试、临时线索、未验证想法先留在本地，不自动上传
部门共享经验已脱敏、可供同类任务检索的经验可以上传，但不是正式制度
企业长期指导反复验证、经过审核的稳定结论进入正式文档并随版本长期保留
第四期会正式讲清三层记忆如何晋升，以及谁决定何时归档。
Course 039 / 13
Evidence Board

证据板让审核人看到事实，而不是相信一句“已经完成”

任务目标

本轮承诺解决什么，不解决什么。

影响范围

改了哪些模块、接口、配置和数据。

测试结果

命令、环境、通过与失败数量、关键输出。

代码差异

新增、删除、复用和偏离计划的内容。

剩余风险

未验证场景、环境缺口和回滚方式。

人工确认

谁确认了范围、架构、测试和合入。

Course 0310 / 13
Human Accountability

AI 可以执行，责任仍然由人承担

角色人的决定不能交给 AI 的责任
需求提出者确认现象、目标和业务优先级AI 不替业务决定做不做
模块负责人确认模块边界、实现质量和兼容性AI 不替负责人批准越界
贡献者检查每个改动文件和真实测试结果AI 生成不等于可以直接提交
审核者根据差异和证据判断是否合入不能只看 AI 摘要
Course 0311 / 13
Risk Prompt

动手前，先让 AI 完成一次团队风险评估

先不要修改代码。请基于源码、Code Graph、项目文档和历史问题评估： 1. 已有能力和复用候选； 2. 可能影响的模块、接口、配置和数据； 3. 是否需要负责人或架构审核； 4. 需要哪些正常、失败和边界测试； 5. 本轮结束后哪些经验只留本地，哪些值得脱敏后共享。
合格输出必须同时说明依据、未知项、人类决策点和验证缺口。
