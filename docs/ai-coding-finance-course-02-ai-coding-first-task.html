第 2 期：让 AI 完成第一次可检查的小改动

认识研发协作词汇，学会让 AI 先读项目、留下任务上下文，再小步修改并提供证据。

本期起点

已经能把一个问题整理成可靠的协作材料。

本期能力

在 AI Coding 工具里完成一次受控的小改动。

为后面埋线

项目知识地图、任务上下文和人工确认点。

学习路径这一段解决什么问题
认识工具Codex、Claude Code、Cursor Agent、Trae 能做什么。
理解项目先读源码、说明和项目结构，再提出计划。
小步修改基于分支执行最小改动，遇到边界停下来。
证明结果用测试、差异和未验证项说明完成情况。
课前准备准备一个可练习的小项目；安装团队支持的任一 AI Coding 工具。
本期操作练习任务只选文字、校验或小逻辑，不碰公共接口和数据库。

Course 022 / 12
Common Words

先把常见研发词翻译成人话

Repository 仓库

项目所有文件及其修改历史所在的地方。

Issue 问题单

记录为什么要改、现象是什么、希望得到什么。

Branch 分支

为本次修改建立的独立工作副本。

PR 合并申请

把改动、证据和风险交给负责人审核。

Test 测试

用可重复方式证明功能符合预期。

Owner 负责人

对模块边界、质量和是否接受改动负责的人。

Course 023 / 12
Safe Loop

第一次改代码，只走这条最小闭环

1理解复述目标，读取相关项目资料。
2计划说明要改哪里、为什么、风险是什么。
3确认人确认范围和基础分支。
4修改只做实现目标所需的最小变化。
5验证运行测试并汇报差异、结果和缺口。
Course 024 / 12
Project Knowledge

AI 理解项目时，需要一张可靠的知识地图

第一期讲“回答要有来源”；进入代码仓后，来源变成源码、结构投影和项目说明。

源码是第一事实

真实类、函数、配置和测试决定系统当前如何运行。

Code Graph 是结构投影

帮助快速看到类、接口、调用和模块依赖，定位影响范围。

文档解释意图

README、架构说明和历史决定解释为什么这样设计。

Issue 提供任务背景

说明本轮为什么改、期望结果和已知限制。

Task Context

每次任务先建立一份可接续的上下文

这份记录连接第一期的工作摘要与后面的团队 Work Context Package。

记录部分至少包含解决的问题
任务issue 地址、目标、完成标准避免改着改着忘记为什么做
项目基础分支、相关模块、关键资料避免读错仓库或改错目录
边界允许修改、禁止修改、需要谁确认避免小需求演变成架构改造
状态已完成、待确认、下一步、证据中断后可以继续
Course 026 / 12
Read First

第一条操作口令：只理解项目，暂时不要改

请先不要修改任何文件。 1. 复述“大文件上传失败”问题和完成标准； 2. 读取仓库入口规则、相关模块 README、源码和测试； 3. 如有 Code Graph，给出相关类和调用链； 4. 列出已有能力、可复用实现、未知项和可能影响的模块； 5. 生成本轮任务上下文摘要，等我确认后再继续。
合格结果不是“我已理解”，而是能指出依据文件、相关模块、可复用实现和未知项。
Course 027 / 12
Plan

第二条操作口令：给出最小计划和停下条件

基于已确认的任务上下文，提出最小修改计划。 请写明：修改文件、复用点、测试、风险、回滚方式。 如果涉及公共接口、跨模块、配置含义或数据库，请停止实现并要求负责人确认。

计划要具体

能看出修改位置、顺序和验证方法。

范围要小

不包含顺手重构、无关清理和另起一套实现。

停下条件要明确

遇到公共边界时把决定交回给人。

后续课程会介绍已经提供的 Compact 模式：低风险小改动可以用一句话启动 Plan 与 Tasks 的联合审核，但仍要分清“技术决定”和“执行步骤”，并由人确认是否适用。
Course 028 / 12
Human Gate

真正修改前，人至少确认三件事

从哪个分支开始

确认基础版本，避免在过期代码上开发。

范围是否正确

确认 AI 找对模块，没有扩大任务。

是否触碰架构边界

公共接口、跨模块和数据归属变化必须升级审核。

AI 可以给建议，但“允许改什么”和“是否接受架构变化”是人的决定。
Course 029 / 12
Implement

第三条操作口令：只做计划内的最小改动

计划已确认。请创建本次任务分支，只修改计划中列出的文件。 优先复用已有上传状态和错误处理；不要新增公共参数或改其他模块数据库。 完成后逐个检查改动文件，并说明任何偏离计划的地方。
做法会发生什么判断标准
新增一套上传状态重复已有能力，后续维护两套逻辑先搜索和复用
顺手重构多个模块范围扩大，审核和回滚困难拆成独立任务
只修改定位到的处理逻辑差异可读、风险可控符合最小变更
Course 0210 / 12
Evidence

完成不是一句话，而是一组可检查证据

改了什么

列出文件和行为变化，并说明为什么需要。

怎么验证

给出真实运行过的命令、测试结果和关键输出。

还没验证什么

明确环境缺口、跳过项和剩余风险。

怎么撤回

说明关闭功能、回滚提交或恢复配置的方法。

证据摘要：修改文件 / 测试命令 / 通过结果 / 未执行项 / 风险 / 回滚方式
Course 0211 / 12
Hands On

课堂练习：完成一次只读分析和最小计划

1选择任务挑一个文字、校验或小逻辑问题。
2建立上下文记录目标、仓库、分支、边界和状态。
3只读分析让 AI 指出源码、测试和复用点。
4审核计划检查范围和停下条件。
5保存结果把上下文和计划留给下一次继续。
