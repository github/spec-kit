Course 041 / 16
SDD

Spec Driven Development：先形成共同理解，再生成代码

1原始需求一句自然语言、业务现象或客户目标。
2Specify明确用户场景、目标、边界和验收标准。
3Plan基于项目结构决定实现方案和影响范围。
4Tasks把计划拆成有顺序、可验证的小任务。
5Implement最小修改、运行验证、形成证据。
文档不是额外负担，而是不同角色、不同 AI 会话之间的交接接口。
Course 042 / 16
Natural Language Entry

用户只需要从一句真实问题开始

我们的大文件上传经常超时，请帮我按团队流程处理。 或者： 我希望管理员可以查看每次上传失败的原因，请帮我启动新特性流程。

AI 先分类

判断是 bug、公开新特性、私有需求还是从零项目。

信息不足先追问

不凭一句话直接修改代码。

生成工作入口

起草 issue、spec 或任务上下文，交给人确认。

Course 043 / 16
Bug Flow

Bugfix：从现象和 coding issue 开始

1报告现象业务人员描述发生了什么，不要求判断根因。
2创建 issue记录复现、影响、期望和缺失信息。
3评估影响定位相关模块和架构风险，由人确认范围。
4修复验证最小修复并补回归、失败路径证据。
5沉淀经验保留根因、误判线索和预防检查项。
Course 044 / 16
Feature Flow

公开新特性：从 coding issue 进入完整 SDD

1提出目标说明用户场景、价值和非目标。
2确认是否做技术委员会或相应决策组织确认方向。
3Specify形成场景、边界和验收标准。
4Plan 与 Tasks架构师规划，开发拆分任务。
5实现与证据代码 PR 关联原始 issue。
Course 045 / 16
Planning Modes

小改动可以少一道流程，但不能少掉思考

团队已经提供 Compact 扩展模式。使用者只需在聊天中明确说出 Compact，AI 就会启动对应流程；它不是让 AI 直接编码，而是把 Plan 和 Tasks 放进一次操作和一次联合审核中。

选择怎么运行适用情况
标准流程Plan 单独审核，再生成 Tasks 并审核新特性、跨模块、公共接口、数据库、安全权限、发布风险
Compact 模式（已支持）一句话启动；架构和开发仍隔离上下文；最后联合审核需求清楚、单模块、路径唯一、失败成本低、容易回滚
例子：给已有 UserValidator 补充空邮箱校验。 Plan：复用现有校验入口，不改变公共接口。 Tasks：补失败测试 → 修改校验 → 运行用户模块自测。 如果分析发现要改公共 API 或数据库，就停止 Compact，回到标准流程。

不是按文件数判断

一个文件也可能是高风险 SPI；多个本地测试文件也可能仍是低风险。

AI 只能建议

先看影响分析，再由负责人确认是否允许使用 Compact。

发现范围扩大就退出

保留已有证据，回到独立 Plan 和 Tasks，不在快捷清单上继续打补丁。

Course 046 / 16
New Project

从零项目：先把地基建好，再开始堆功能

它与新特性共享 SDD 主流程，但要额外明确产品边界、架构骨架和第一条可运行链路。

产品基线

服务谁、解决什么、不解决什么、第一版成功标准。

架构基线

模块职责、公共接口、数据归属、依赖最小集。

工程基线

目录、构建、测试、日志、配置和发布方式。

运行主干

先完成一条端到端可运行的小链路，再扩展能力。

Course 047 / 16
Role Isolation

三个 AI 角色隔离上下文，用文档或 issue 交接

角色交付物上下文边界
产品/客户经理角色specify：用户场景、目标、边界、验收不带入实现偏好
架构师角色plan：模块、接口、复用、风险和验证策略基于 spec 和项目知识判断
开发角色tasks + implement：拆任务、编码、自测和证据不擅自改写需求和架构决定
上下文隔离让责任更清楚，也减少同一个 AI 为自己早先的假设辩护。
Course 048 / 16
Work Context

Work Context Package 让任务可以中断和继续

文件作用下次打开时先看什么
context-pack.md面向下一位协作者的任务摘要目标、来源、模块、边界、证据要求
work-context.yml机器可读的当前状态工作标识、状态、URL、下一步和停下条件
evidence/本轮验证材料索引命令、结果、环境、失败和未验证项
state.json工作流引擎的执行进度用于从暂停的步骤继续运行
这些文件是否进入 Git 按项目策略决定；本地敏感内容和临时运行状态默认不上传。
Course 049 / 16
Code Graph

Code Graph 在计划前帮助 AI 看清影响范围

上传页面 → UploadController → UploadService → StorageAdapter ↘ UploadEvent → AuditService 计划问题：新增“失败原因”会影响返回对象、事件结构、审计记录和哪些测试？

先定位

找出相关类、接口、调用链和数据流。

再复用

识别已有能力和扩展点，避免重复实现。

控制半径

优先单模块修改；跨模块时明确接口需求和负责人。

Course 0410 / 16
Three Support Layers

Skill、Knowledge、Memory 分别解决三类问题

外围支撑回答的问题典型内容
Skill这一步怎么做启动任务、影响分析、审核、验证、提交和复盘流程
Knowledge当前项目是什么源码、Code Graph、术语、架构、接口和历史决定
Memory过去发生过什么已做决定、失败尝试、bug 根因和可复用经验
三层共同作用：Skill 决定动作，Knowledge 提供事实，Memory 提醒历史经验和已知风险。
Course 0411 / 16
Three Memory Tiers

三级记忆按敏感度、成熟度和长期价值管理

记忆层保存内容Git 与发布边界
本地记忆个人尝试、临时线索、未验证判断不上传；可随任务清理
部门记忆脱敏后的案例、排障经验、可检索模式上传内部空间；不进入正式 docs 和版本记录
企业记忆稳定架构原则、成熟实践、长期兼容与运维经验审核后进入 docs；随正式版本长期保留
Course 0412 / 16
Memory Consolidation

经验归档不必等发布，由贡献者和维护者按价值触发

1选择来源完成的 feature、bugfix、事故、迁移或评审意见。
2提炼经验保留问题、判断条件、做法、证据和适用范围。
3隐私处理删除客户、账号、内部地址和不可公开实现细节。
4选择层级本地、部门或企业；不默认全部晋升。
5审核与复用进入检索、规则、测试或正式文档。
发布归档是批量整理的一种场景，不是每次发布前必须完成的硬 Gate。
Course 0413 / 16
Human Gates

关键节点由人做决定，Evidence Board 提供依据

人工确认点主要判断典型责任人
Spec Gate目标、边界和验收是否正确需求提出者或相应负责人
Plan Gate架构、复用、影响半径和验证策略是否合理架构师、模块负责人
Task Gate任务是否可执行、顺序正确、没有遗漏验证维护者、贡献者
PR Gate差异与证据是否足以合入代码审核者、测试和运维责任人
Course 0414 / 16
Private Requirement Extension

扩展场景：私有客户需求只把可公开的交接信息带入代码仓

1内部留痕原始企业需求记录在严格权限的内部需求仓。
2审核脱敏形成可供研发使用的安全摘要和验收边界。
3URL 交接代码仓通过需求 URL 关联来源，不复制原始私密内容。
4Plan 融合计划阶段获取内容，生成本地忽略的 spec.override.md。
5公开实现代码、测试和 PR 只保留实现所需的公开安全信息。

Final Practice

毕业练习：用一句话启动，并在任意 Gate 继续

我们的大文件上传经常超时，请按团队 bugfix 流程处理。 如果信息不足，先帮我起草 issue 并列出待确认项；不要猜根因。 继续任务时： 请恢复“bug-upload-timeout-123”的 Work Context，先告诉我当前阶段、已有决定、失败尝试、证据缺口和下一步，等我确认后继续。

你负责目标

确认真实问题、业务边界和是否继续。

AI 负责执行

读取知识、遵循 Skill、记录上下文、生成改动和证据。

团队负责质量

审核架构、验证结果、合入代码并沉淀经验。
