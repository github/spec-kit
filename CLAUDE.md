# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Spec-Kit is a toolkit for **Spec-Driven Development (SDD)** - a methodology where specifications become executable and directly generate working implementations. The project provides:

1. **Specify CLI** (`src/specify_cli/`) - A command-line tool to bootstrap new SDD projects with appropriate templates and slash commands
2. **MCP Server** (`src/speckit_mcp/` and `spec-kit-mcp-server.py`) - Model Context Protocol server providing tools for specification, planning, domain analysis, and task management
3. **Templates and Scripts** (`.specify/`, `templates/`) - Reusable templates and shell scripts for AI agent commands

## Development Commands

### Building and Packaging

```bash
# Run tests
pytest tests/ -v

# Run specific test file
pytest tests/test_domain_analysis.py -v

# Run tests with coverage
pytest tests/ --cov=src/specify_cli/ --cov-report=html

# Test the MCP server directly (safe testing via test harness)
python test_mcp_server.py

# Build the specify-cli package
python -m build

# Install locally in editable mode
pip install -e .
```

### Testing the CLI

```bash
# Check for installed tools
specify check

# Initialize a new project (interactive)
specify init my-test-project

# Initialize with specific AI assistant
specify init my-test-project --ai claude

# Initialize in current directory
specify init . --ai claude

# Show version
specify version
```

### Release Process

Releases are automatically created when pushing to `main` branch. The `.github/workflows/release.yml` workflow:
1. Determines next version from git tags
2. Creates release packages for each AI assistant (claude, copilot, cursor, etc.) and script type (sh/ps)
3. Generates release notes
4. Creates GitHub release with assets

## Architecture

### Specify CLI (`src/specify_cli/`)

The main CLI entry point is `src/specify_cli/__init__.py` (54KB, ~1350 lines). Key components:

- **Agent Configuration**: `AGENT_CONFIG` dict defines supported AI assistants (copilot, claude, gemini, cursor-agent, qwen, opencode, codex, windsurf, kilocode, auggie, codebuddy, roo, q, amp, shai) with their folder paths and CLI requirements
- **Download & Extract**: `download_and_extract_template()` fetches release assets from GitHub and extracts them
- **Interactive Selection**: `select_with_arrows()` provides Rich-based TUI for agent/script selection
- **VS Code Settings Merging**: `merge_json_files()` handles merging `.vscode/settings.json` when templates are applied to non-empty directories
- **Step Tracker**: `StepTracker` class renders hierarchical progress with Rich Live display

**Special handling for Claude CLI**: The `check_tool()` function checks `~/.claude/local/claude` path first (after `claude migrate-installer` removes the executable from PATH).

### MCP Server Architecture

Two implementations exist:

1. **`spec-kit-mcp-server.py`** (41KB, root directory) - Standalone MCP server using `mcp.server.Server` with tool registration via `@server.call_tool()`
2. **`src/speckit_mcp/server.py`** - FastMCP-based implementation (if FastMCP is available)

Tools provided (from `src/speckit_mcp/tools/`):
- `specify_tool` - Create feature specifications
- `plan_tool` - Generate implementation plans
- `tasks_tool` - Break down plans into tasks
- `analyze_domain_tool` - Extract business entities from data files
- `clarify_tool` - Clarify ambiguous requirements
- `constitution_tool` - Manage project principles

### Domain Analysis Module

`src/specify_cli/domain_analysis.py` analyzes JSON/CSV data files to extract:
- Business entities with field types and relationships
- Business rules derived from data patterns
- Integration points with external systems

Supports multiple domain types: financial, ecommerce, crm

### Template System

Templates are stored in:
- `.specify/templates/` - Project-local templates after initialization
- `templates/` - Repository source templates

Key templates:
- `spec-template.md` - Feature specification with user stories, requirements, success criteria
- `plan-template.md` - Implementation plan with technical architecture
- `tasks-template.md` - Task breakdown structure

### Slash Command Structure

Each AI assistant has its own slash command format:
- **Claude Code**: `.claude/commands/speckit.*.md` files
- **GitHub Copilot**: `.github/prompts/speckit.*.prompt.md` files
- **Cursor**: `.cursor/commands/speckit.*.md` files

Commands are generated by the release workflow scripts in `.github/workflows/scripts/create-release-packages.sh`.

## Spec-Driven Development Workflow

1. **`/speckit.constitution`** - Establish project principles and quality standards
2. **`/speckit.specify`** - Create feature specification from user description (creates Git branch)
3. **`/speckit.clarify`** (optional) - Ask structured questions to resolve ambiguities
4. **`/speckit.plan`** - Generate technical implementation plan with architecture decisions
5. **`/speckit.tasks`** - Convert plan into executable task list
6. **`/speckit.analyze`** (optional) - Cross-artifact consistency analysis
7. **`/speckit.checklist`** (optional) - Generate quality validation checklist
8. **`/speckit.implement`** - Execute tasks to build the feature

## Environment Variables

- `GH_TOKEN` / `GITHUB_TOKEN` - GitHub token for API requests (increases rate limits from 60/hour to 5000/hour)
- `SPECIFY_FEATURE` - Override feature detection for non-Git repos (set to feature directory name)

## Important Patterns

### Script Permissions

The `ensure_executable_scripts()` function in `src/specify_cli/__init__.py` recursively sets execute permissions on `.sh` scripts in `.specify/scripts/` (POSIX only, no-op on Windows).

### Git Branch Naming

Feature branches follow pattern: `[###-feature-name]` where `###` is auto-incremented feature number (001, 002, 003, etc.)

### Template Placeholder Markers

Templates use `[NEEDS CLARIFICATION: ...]` markers for ambiguous requirements that must be resolved before implementation planning.

### CLI Tool Detection

When checking for AI agent CLI tools, some agents are IDE-based (copilot, cursor-agent, windsurf, kilocode, roo) and don't require CLI checks. Others (claude, gemini, qwen, opencode, codex, auggie, codebuddy, amp, shai) require CLI verification via `shutil.which()`.

## Testing Notes

- MCP servers are long-running processes; avoid running them directly in main process
- Use `timeout 5s python spec-kit-mcp-server.py` for quick validation
- Run in tmux or background for manual testing
- Tests use sample data in `sample-data/` and `tests/data/sample/`
