## Executive Summary

Интеграция проектов (SpecKit, AgentForge patterns, AgentMemory-MCP, SkillsMP) для создания глобальной системы памяти агентов с автоматическим накоплением знаний по проектам, поиском существующих скиллов через skillsmp.com (web scraping), и многоуровневой векторной памятью.

**Ключевая проблема**: При разработке агентов и скиллов отсутствует:
1. Централизованная память агентов между сессиями
2. Поиск существующих решений перед созданием новых скиллов (через skillsmp.com web scraping)
3. Автоматическое накопление знаний по каждому проекту
4. Унифицированная система для создания и управления агентами

**Решение**: Создать глобальную интеграцию SpecKit в `C:\Users\TaoBao\.claude` с:
- Поиском скиллов через **skillsmp.com** (400K+ скиллов, web scraping)
- Векторной памятью через agent-memory-mcp с **mxbai-embed-large**
- Автоматической RAG индексацией через Ollama при изменении файлов
- Паттернами создания агентов от AgentForge (без интеграции с OpenClaw)

### FR2: Система памяти по проектам

- **FR2.1**: Структура `C:\Users\TaoBao\.claude\memory\projects\{project-name}\`
  - `lessons.md` - накопленные уроки
  - `patterns.md` - паттерны улучшений
  - `projects-log.md` - история задач
  - `architecture.md` - архитектурные решения
  - `handoff.md` - текущий контекст сессии
- **FR2.2**: **Изоляция файловой памяти по проектам**:
  - Каждый проект имеет собственную файловую память
  - Агент в сессии проекта А не видит файловую память проекта Б
  - При переключении проекта - загружается память только этого проекта
- **FR2.3**: **Перекрёстная векторная память**:
  - Общая векторная индексация для всех проектов
  - Поиск best practices и решений ошибок из других проектов
  - Агент понимает источник памяти (source_project)
  - **Механизм перекрёстного обновления**:
    - При сохранении: контекст содержит project_id для обновления
    - Явное указание в MemoryEntry.related_projects: ["spec-kit", "agent-memory-mcp"]
    - Валидация: проверка существования .spec-kit/project.json
    - Конфликты: last write wins с timestamp
- **FR2.4**: **Определение активного проекта**:
  - Активный = текущая рабочая директория проекта
  - **project_id определяется из .spec-kit/project.json**:
    - Конфиг создаётся автоматически при первом запуске
    - Формат: {"project_id": "spec-kit", "project_name": "SpecKit", "memory_enabled": true}
    - Возможность явного указания при необходимости
  - При переключении директории → загрузка файловой памяти нового проекта
  - Векторная память остаётся общей и доступной для поиска
- **FR2.5**: **Умный поиск (scope auto-determination)**:
  - **Явные маркеры** (переопределяют всё):
    - Глобальный: "вообще", "везде", "в других проектах", "во всех"
    - Локальный: "здесь", "в этом проекте", "локально", "у нас"
  - **Классификация intent** (если нет явных маркеров):
    - Глобальный: "как исправить", "как решить", "best practice", "оптимальный подход"
    - Локальный: "где находится", "как мы делали", "наш код", "наша база"
  - **Fallback**: если локальный < 3 результатов → автоматически глобальный
  - **Самообучение**:
    - Запись effectiveness результата (пользователь принял или нет)
    - Корректировка правил на основе успешных/неуспешных поисков
    - Накопление паттернов в patterns.md для scope determination

### FR4: Интеграция agent-memory-mcp

- **FR4.1**: Установка и настройка agent-memory-mcp
- **FR4.2**: Ollama интеграция с `mxbai-embed-large`:
  - Модель: `mxbai-embed-large` (1024d)
  - Эндпоинт: `http://localhost:11434`
  - Локальный режим только
  - **Graceful degradation при недоступности Ollama**:
    - **Проверка при запуске агента**:
      - Проверка MCP сервера `agent-memory-mcp` (installed/enabled)

      - Файловая память: **работает полностью** (lessons.md, patterns.md, etc.)

      - RAG индексация: **пропускается**, никаких ошибок
      - Поиск: **только grep по файловой памяти**
    - **Никаких исключений**: система продолжает работу нормально
    - **Перепроверка**: опционально при следующем запуске агента
- **FR4.3**: 4 типа памяти:
  - `episodic` - события, эксперименты

  - `working` - текущий контекст
- **FR4.4**: Автоматическая RAG индексация:
  - Watch mode для отслеживания изменений файлов
  - Документы: `README.md`, `docs/`, `CHANGELOG.md`
  - Код: выборочно по важности

  - Структурированное резюме при завершении задачи:

    - **Related Projects**: какие проекты затронуты (для перекрёстных ссылок)
    - **Tags**: keywords для семантического поиска
  - Векторизация: весь markdown текст через mxbai-embed-large
  - Metadata: project_id, task_id, timestamp, source_project
  - Инкрементальное обновление через Ollama

### FR11: Контекстная оптимизация (Memory-Aware Workflow)

- **FR11.1**: **Hybrid стратегия чтения памяти**:
  - **Lightweight (всегда, ~80-120 tokens)**:

  ```

### FR13: Auto-Documentation Pattern с AI-классификацией

- **FR13.1**: **Сохранение при завершении задачи** (всегда):
  - Структурированный отчет в векторную память
  - Копия в projects-log.md для истории
  - Формат:
    ```markdown
    ## Problem: [что решали]
    ## Solution: [как решили - код/подход]
    ## Lessons Learned: [что узнали для будущего]
    ## Related Projects: [какие проекты затронуты]
    ## Tags: [ключевые слова для поиска]
    ```
- **FR13.2**: **AI-классификация для определения важности**:
  - **Метод**: Анализ содержания + контекст для определения важности
  - **Факторы для AI-анализа**:
    - **Семантическая важность**: слова "архитектура", "дизайн", "решение", "выбрали"
    - **Контекстная сложность**: длина обсуждения, количество альтернатив
    - **Техническое влияние**: затрагивает ли ключевую архитектуру
    - **Повторяемость**: похожие решения в других контекстах
  - **AI-скоринг**: 0.0-1.0 (importance score)
    - >0.7: Высокая важность → architecture.md + векторная память
    - 0.4-0.7: Средняя важность → patterns.md (если повторяется)
    - <0.4: Базовое → только projects-log.md
- **FR13.3**: **Явные маркеры пользователя** (переопределяют AI):
  - "документируй это в architecture.md"
  - "запомни это как паттерн"
  - "ошибка X важна, запиши"
  - Явный маркер → importance = 1.0 (максимальная)
- **FR13.4**: **Авто-детекция через AI**:
  - **Ключевые слова**: "решение", "выбрали", "архитектура", "дизайн", "подход"
  - **Наличие артефактов**: diagram, schema, design document, code snippet
  - **Длина обсуждения**: >3 раундов = повышенный приоритет
  - **Альтернативы**: рассматривалось >1 варианта = важное решение
- **FR13.5**: **Куда сохранять** (на основе AI-классификации):
  - **architecture.md**: importance >0.7 ИЛИ явный маркер
  - **lessons.md**: была ошибка ИЛИ исправление бага
  - **patterns.md**: importance 0.4-0.7 ИЛИ 3+ повторений
  - **projects-log.md**: всегда (все задачи)
- **FR13.6**: **Автоматический паттерн-детектор**:
  - Отслеживать повторяющиеся решения
  - При 3+ повторениях → предложение сохранить как паттерн
  - Сохранить в patterns.md с тегом #autodetected
  - Пользователь может отредактировать/удалить

## Constraints

### C1: Локальный режим
- По умолчанию: только локальное хранилище
- Опционально: облачные embeddings (Jina AI, OpenAI)

### C2: Независимая система
- БЕЗ интеграции с OpenClaw ecosystem
- Используются только паттерны и шаблоны от AgentForge
- Собственный формат агентов

### C3: Обратная совместимость
- Существующие проекты должны работать без изменений
- Миграция должна быть необязательной

### C4: Единство инсталляции
- SpecKit обновляется один раз в `F:\IdeaProjects\spec-kit`
- Все проекты используют симлинк для доступа

### C5: Сохранение по задачам
- Память сохраняется при завершении каждой задачи
- Потеря данных при сбое недопустима

### C6: Web scraping ограничения
- Rate limiting для requests к skillsmp.com
- Локальный кэш для работы оффлайн
- Уважение robots.txt и terms of service

### C7: Фиксированная модель embeddings
- Используется только `mxbai-embed-large`
- Избежание model mismatch проблемы
- Все embeddings генерируются одной моделью

### C8: Разрешение конфликтов скиллов
- При конфликте - запрос пользователя с рекомендацией
- Автоматическая замена без согласования НЕ допускается
- Пользователь имеет финальное решение

### C9: Автоматическая RAG индексация
- Watch mode для отслеживания изменений файлов
- Debounce для rapid changes
- Инкрементальные обновления (только изменённые файлы)
- Исключение чувствительных файлов (.env, secrets, etc)

### C10: Изоляция и перекрёстная память
- **Файловая память**: строго изолирована по проектам
- **Векторная память**: общая для всех проектов, но с понятным source_project
- **Активный проект**: определяется текущей рабочей директорией
- **Перекрёстное обновление**: решение в проекте B МОЖЕТ обновить векторную память проекта А
- **Перекрёстный поиск**: только для best practices и решений ошибок

### C11: Graceful Degradation
- **Система продолжает работать при недоступности внешних зависимостей**
- Ollama недоступен → файловая память работает, разовое предупреждение
- agent-memory-mcp недоступен → fallback на grep поиск
- SkillsMP недоступен → система работает без поиска скиллов
- Никаких exceptions/критических ошибок, только warnings
- Проверки при запуске агента, разовые уведомления

## Risks & Mitigation

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| SkillsMP изменит HTML структуру | Средняя | Среднее | Локальный кэш + versioned scrapers |
| SkillsMP заблокирует scraping | Средняя | Высокое | Локальный кэш + fallback на GitHub поиск |
| Ollama недоступен | Средняя | Низкое | **Graceful degradation**: проверка MCP при старте, разовое warning, файловая память работает, система без ошибок |
| mxbai-embed-large недоступна в Ollama | Низкая | Среднее | Fallback на bge-m3 |
| Проблемы с model mismatch | Низкая | Низкое | Фиксированная модель исключает проблему |
| Частые изменения файлов → нагрузка на Ollama | Средняя | Среднее | Debounce + batch processing |
| Конфликты между проектами | Низкая | Среднее | Изоляция файловой памяти по project_id |
| Симлинк не работает на Windows | Низкая | Среднее | Fallback на копию или PATH |
| Потеря данных при сбое задачи | Низкая | Высокое | Atomic writes + WAL mode SQLite |
| Чувствительные данные попадут в индекс | Средняя | Высокое | `.gitignore` + исключения по расширению |

## Research Notes

### Найден хаб скиллов: SkillsMP.com

**Критическое находка**: https://skillsmp.com/ - marketplace для agent skills с 400K+ скиллов!

**Возможности**:
- Поиск с AI семантикой или ключевыми словами
- Категоризация по use case
- Сортировка по популярности
- Все скиллы используют SKILL.md standard
- Совместимость с Claude Code, OpenAI Codex CLI, ChatGPT

**Интеграция** (исследование 2025-03-10):
- **Официальный API: НЕТ** (исследование подтвердило отсутствие API)
- Используется **web scraping** HTML страниц
- Локальный кэш для оффлайн работы
- Прямые ссылки на GitHub репозитории

### Выявленные грабли (из Habr статьи)

**Model Mismatch Problem**:
- Вектора от разных моделей (Jina, OpenAI, Ollama) несовместимы
- Косинусное расстояние между векторами разных моделей = случайное число
- **Решение**: использовать только одну модель (mxbai-embed-large) для исключения проблемы

**Ограничения in-memory поиска**:
- O(n) поиск работает до десятков тысяч записей
- Для >100K нужен ANN-индекс (HNSW, IVF)
- **Решение**: начать с in-memory, мониторить производительность

**Проблемы fallback**:
- Fallback между провайдерами embeddings опасен
- Может дать "уверенно неправильные" результаты
- **Решение**: local-only режим с фиксированной моделью

### AgentForge паттерны (без интеграции с OpenClaw)

**Используемые компоненты**:
- 4-уровневая память (контекстная, файловая, векторная, identity)
- Система автоулучшения (ошибка → паттерн → правило)
- Шаблоны файлов агента (AGENTS.md, SOUL.md, USER.md, MEMORY.md)
- Auto Handoff механизм

**НЕ используется**:
- OpenClaw gateway интеграция
- OpenClaw конфигурация
- OpenClaw agent-to-agent communication
